## アプリの概要

Handover Notifier は、大学サークルの役職引き継ぎを支援する通知管理Webアプリ。ユーザーがWebアプリから通知内容と送信日を登録すると、Google Apps Script がスケジュールを確認し、指定日の朝に Slack または Gmail へ通知を送る。

スライドでは、以下のように説明できる。

- 引き継ぎタスクをWebアプリから登録できる。
- 登録した送信日に Slack または Gmail へ通知できる。
- 必要なタイミングで引き継ぎ内容を思い出しやすくする。
- 前年度の通知予定を次年度向けに再利用できる。

## このアプリでできること

- 通知の追加、編集、削除
- 通知タイトル、本文、送信日、送信先、参考リンクの登録
- Slack または Gmail の送信先選択
- 設定画面で Slack Incoming Webhook URL と Gmail 送信先メールアドレスを保存
- 通知カードからの「今すぐ送信」
- 前年度スケジュールからの複製
- 全データ削除

「前年度スケジュールからの複製」は、過去年度データを別に増やす機能ではなく、既存通知の送信日を +1 年し、送信済み状態をリセットする年度更新操作として扱う。

通常ユーザーの主な操作場所はWebアプリ。Google Sheets はデータ保存先だが、スライドでは通常ユーザーが直接シートを編集するように見せない。

## 通知登録の流れ

実アプリの追加・編集フォームでは、以下を入力する。

| 項目 | 内容 |
|---|---|
| 通知メッセージ | 通知本文。何をすべきかを具体的に書く。 |
| タイトル | 一覧表示や通知件名で使う表示名。 |
| 送信日 | この日の朝に自動送信される。 |
| 送信先 | Slack または Gmail を選ぶ。 |
| 参考リンク | 任意。必要な場合だけ添付する。 |

保存時には、タイトル・メッセージ・送信日が必須。送信先の詳細情報は通知ごとには入力せず、設定画面で管理する。

## Slack通知の仕組み

Slack通知は Slack Incoming Webhook を使う。

- 設定画面で Slack Incoming Webhook URL を保存する。
- 通知の `dest` が `slack` の場合、GAS が Webhook URL にJSONをPOSTする。
- Slackに送る本文は、タイトル、メッセージ、参考リンクがある場合はリンクを含む。
- 送信先チャンネルは Webhook URL 作成時に決まる。
- 通知ごとにSlackチャンネルを切り替える機能はない。

スライドでは「Slackに通知できる」「Slackチャンネルへ届く」と表現してよいが、「通知ごとにチャンネルを選べる」とは言わない。

## Gmail通知の仕組み

Gmail通知は Google Apps Script の `MailApp.sendEmail()` を使う。

- 設定画面で Gmail 送信先メールアドレスを保存する。
- 複数メールアドレスはカンマ区切りで保存される。
- 通知の `dest` が `email` の場合、登録済みメールアドレスへ送信する。
- 件名は `[引き継ぎ通知] ` + 通知タイトル。
- 本文は通知メッセージ、参考リンクがある場合はリンクを含む。

スライドでは「Gmailに通知できる」「メールで受け取れる」と表現してよい。

## Apps Scriptの役割

Google Apps Script は、Webアプリ表示、API、データ保存、通知送信、定期実行を担当する。

- `doGet()` でWebアプリ画面を返す。
- `apiSaveNotification()` などのAPIでフロントエンド操作を受け付ける。
- 初回セットアップの `initialize()` でシート作成と日次トリガー登録を行う。
- `dailyTrigger()` が毎朝8時に実行される。
- 当日の未送信通知をSlackまたはGmailへ送る。
- 送信成功後に `sent` を `true` にする。
- 送信失敗時はログ出力し、デプロイ者宛にエラーメールを送る。

スライドでは、Apps Script の詳細説明は控えめにし、「毎朝チェックして、当日の通知を送る役割」と伝えるのがよい。

## Google Sheetsの役割

Google Sheets はデータ保存先として使われる。

`notifications` シートには以下の列を持つ。

- `id`
- `title`
- `message`
- `date`
- `dest`
- `link`
- `sent`

`settings` シートには以下の設定値を保存する。

- `slackWebhookUrl`
- `gmailAddresses`

スライドでは「Google Sheetsに保存される」と内部説明として扱ってよい。ただし、通常ユーザーが日常的にGoogle Sheetsを直接編集するようには見せない。

## 年度更新・前年度スケジュール再利用

ホーム画面には「前年度スケジュールから複製」機能がある。

実装上は、全通知に対して以下を一括適用する。

- 送信日を +1 年する。
- `sent` を `false` にリセットする。
- タイトル、メッセージ、参考リンクは変更しない。

この機能は、役職交代や新年度の引き継ぎで、昨年の予定をベースにして必要な通知だけ後から編集する運用を想定している。

注意点として、実装は「既存の全通知を更新する」挙動であり、過去年度データを別シートに保存したり、複数年度を並行管理したりする設計ではない。

## 正しく言える表現

- Webアプリから通知を登録できます。
- 指定日の朝に Slack または Gmail へ通知できます。
- Apps Script が毎朝、当日の未送信通知を確認します。
- 通知後は送信済みとして扱われます。
- 前年度の予定をもとに、日付を1年進めて再利用できます。
- 引き継ぎタスクを思い出しやすくします。
- 抜け漏れを防ぎやすくします。
- 資料を毎回探しに行く負担を減らせます。

## 避けるべき表現

以下は、機能の誤解につながるためスライドで使わない。

- 予定変更を自動検知する
- 見逃しゼロ
- 抜け漏れが完全になくなる
- Google Calendarと連携する
- LINEに通知できる
- 通知ごとにSlackチャンネルを自由に選べる
- 普通の利用者がGoogle Sheetsを直接編集して使う
- 複数年度のデータを自動で管理する
- AIが引き継ぎ内容を自動生成する
- 既存資料の更新を自動で読み取る
- 企業向けの重い表現として「リードタイム」「生産性を底上げ」などを強調する

## スライドでの説明の軸

スライドでは、技術詳細よりもサークルメンバーの体験を中心にする。

- 資料を探しに行く方式から、必要な日に通知が届く方式へ変える。
- 後輩が常に引き継ぎ資料を気にし続ける負担を減らす。
- 通知タイトル、本文、送信日、送信先を登録するだけで使える印象にする。
- Slack/Gmail、Apps Script、Google Sheets は「裏側の仕組み」として簡潔に扱う。
