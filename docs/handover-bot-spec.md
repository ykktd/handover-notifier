# 引き継ぎ通知Bot — 実装仕様書

## プロジェクト概要

大学サークルの役職引き継ぎを支援するプッシュ型通知システム。
Google Apps Script (GAS) + Google スプレッドシートで実装する。
非エンジニアの後任者が単独で運用・引き継ぎできることを最優先とする。

---

## 技術スタック

| 項目 | 選定 |
|------|------|
| バックエンド | Google Apps Script (GAS) |
| DB | Google スプレッドシート（コンテナバインド） |
| フロントエンド | GAS Web App（HTML/CSS/JS） |
| 定期実行 | GAS 時間主導型トリガー（1日1回） |
| 通知チャンネル | Slack Incoming Webhook / Gmail |
| 開発環境 | clasp（ローカル開発） |

---

## データモデル

スプレッドシートの1シート（`notifications`）で全データを管理する。
年度をまたいで複数シートを持たない。常に「現役データ」1本のみ。

### カラム定義

| カラム名 | 型 | 説明 |
|----------|----|------|
| id | string | UUID など一意な識別子 |
| title | string | 通知のタイトル（一覧表示用） |
| message | string | 通知本文。テキストのみで内容が完結することを想定 |
| date | string | 送信予定日（YYYY-MM-DD） |
| dest | string | `"slack"` または `"email"` |
| sent | boolean | 送信済みフラグ |

### 設定データ

別シート（`settings`）にキーバリュー形式で保存する。

| キー | 値 |
|------|----|
| slackWebhookUrl | Slack Incoming Webhook URL |
| gmailAddresses | 送信先メールアドレス（複数の場合はカンマ区切り） |
| roleName | 役職名（任意）。設定すると一覧画面のタイトル横に「（役職名）」として表示される |

## 機能仕様

### 1. 定期自動送信

- GAS の時間主導型トリガーで1日1回実行
- `date` が当日かつ `sent === false` の通知を抽出して送信
- 送信成功後に `sent` を `true` に更新

### 2. 今すぐ送信

- 管理UI上の任意の通知に対して手動で即時送信できる
- `sent` フラグの状態に関わらず送信可能
- 送信後は `sent` を `true` に更新する
- 用途：設定内容の確認、臨時の手動通知

### 3. 前年度スケジュールの複製（年度引き継ぎ）

- 実行すると全通知に対して以下を一括適用する
  - `date` を +1年（例：2026-03-01 → 2027-03-01）
  - `sent` を `false` にリセット
- メッセージ・タイトルは変更しない
- 複製後、変更が必要な通知のみ個別に編集する運用を想定

### 4. 通知の CRUD

- 管理UI から通知の追加・編集・削除を行う
- タイトル・メッセージ・送信日は必須項目

### 5. 送信先設定

- Slack Webhook URL と Gmail 送信先アドレスを管理UI から設定・保存する
- 各通知では `dest`（slack / email）の選択のみを行う
- 送信先の詳細（どのチャンネルか・誰に送るか）は設定画面で一元管理する
- **Slack の送信先チャンネルは Webhook URL 作成時に固定される。通知ごとにチャンネルを指定する機能は不要**
- 役職名（任意）を登録すると、一覧画面のタイトル横に表示され、どの役職用のBotか判別できる

## 管理UI 画面構成

### 画面①：通知一覧（ホーム）

- 全通知を送信月ごとにグルーピングして表示
- 各カードに表示する情報：タイトル、メッセージ冒頭プレビュー（〜75文字）、送信日、送信先アイコン（Slack / Gmail）
- 各カードのアクション：今すぐ送信、編集、削除
- 画面下部に「前年度スケジュールから複製」ボタンを配置

### 画面②：通知 追加 / 編集フォーム

- 入力項目（上から順）：
  1. 通知メッセージ（主役。最上部・最大サイズのテキストエリア。文字数カウンター付き）
  2. タイトル
  3. 送信日
  4. 送信先（Slack / Gmail のセグメントボタン）
- バリデーション：タイトル・メッセージ・送信日は必須
- 編集時は削除ボタンを下部に表示

### 画面③：設定

- 役職名の入力・保存（任意。設定すると画面①のタイトル横に表示される）
- Slack Incoming Webhook URL の入力・保存
- Gmail 送信先メールアドレスの追加・削除・保存（複数登録可）
- データ管理セクション：全データ削除（危険操作として視覚的に分離）

### 画面④：今すぐ送信モーダル

- 一覧画面の各カードから呼び出す
- 表示内容：送信先バッジ（Slack / Gmail）、通知タイトル、メッセージ全文、リンク（あれば）
- 画面中央に表示（ボトムシートではない）
- 「今すぐ送信」ボタン押下後、完了フィードバックを表示

## 設計上の決定事項と理由

| 決定事項 | 理由 |
|----------|------|
| LINE非対応 | Messaging API のセットアップコストが高く、非エンジニアの後任者が独立して運用するという目標に合わない |
| Webhook URL をUI から設定する | スクリプトプロパティよりも後任者が自分でセットアップできる。Webhook URL の漏洩リスクは「そのチャンネルへの投稿が可能になる」のみであり、サークル内のアクセス権限管理の範囲で許容できる |
| データは常に1本管理（複数年度シートを持たない） | 過去年度を参照するユースケースより、「今年のデータをすぐ使える」ことを優先。複雑なシート管理を後任者に求めない |
| メッセージをフォームの主役にする | リンク添付なしでテキストのみで内容が完結する通知が多い。設計でそれを促すことで通知品質を担保する |
| 送信先チャンネルは通知ごとに指定しない | 送信先チャンネルは役職ごとに固定されるケースがほとんど。通知ごとに指定させると設定漏れ・誤送信のリスクが増す |
| Web App のアクセス設定は `ANYONE_ANONYMOUS` + `executeAs: USER_DEPLOYING` | サークル内の複数人（役職交代後の新旧担当者など）が CRUD を行えるようにする必要がある。URL から管理画面に入れる設計にすることで、共同編集者をスプレッドシートに招待する手間を省きつつ、メール送信元アドレスもデプロイ者で固定できる。URL を Slack 限定公開チャンネル等に置く前提で運用する |

---

## 非機能要件

- プログラミング知識のない後任者が単独でセットアップ・運用できること
- 引き継ぎコストを最小化すること（複製機能による省力化）
- 団体ごとに独立したGoogleドライブフォルダで完結する構成とする
- 日本語UIのみ対応
